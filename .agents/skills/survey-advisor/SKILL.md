# Survey Advisor — PM 유저 리서치 설문 평가 스킬

PM이 유저 리서치 목적으로 설문조사를 설계하거나 기존 질문을 검토할 때 활용하는 스킬.
설문의 구조적 완성도, 질문 품질, 가설 정합성을 평가하고 개선안을 제안한다.

---

## 평가 기준 6가지 (체크리스트)

설문 또는 질문을 검토할 때 아래 6가지 기준을 모두 적용한다.

### 1. 구체적 예시 포함 여부 및 선택지 품질

질문 자체 또는 안내문에 응답자가 자신의 경험에서 즉시 떠올릴 수 있는 구체적 예시가 있어야 한다.

- 나쁜 예: "평소 어떤 방식으로 업무를 관리하나요?"
- 좋은 예: "평소 어떤 방식으로 업무를 관리하나요? (예: 노션에 할 일 목록을 작성한다, 슬랙 메시지를 별표 표시해둔다, 머릿속으로만 기억한다 등)"

선택형 문항의 경우, 선택지 자체의 품질도 함께 평가한다.

- 상호배타성: 선택지 간 중복이 없는가?
- 포괄성: "기타" 포함 여부, 실제 응답 경향을 커버하는가?
- 중립성: 특정 선택지가 더 바람직해 보이도록 표현되지 않았는가?

### 2. 응답자가 자신의 경험과 생각에서 답을 끌어올 수 있는가

질문이 추상적인 의견을 묻지 않고, 응답자의 실제 행동이나 경험을 회상하게 유도해야 한다.

- 피해야 할 패턴: "~하는 것이 중요하다고 생각하나요?" (동의 편향)
- 권장 패턴: "최근에 ~한 상황을 떠올려보세요. 그때 어떻게 하셨나요?"

### 3. 예상 답변과 의사결정 시나리오

각 질문에 대해 예상되는 답변 경향(A형/B형 등)과 그에 따른 의사결정 방향이 설계되어 있어야 한다.
평가 시, 다음 질문을 던진다:

- 이 질문의 답변이 X라면 어떤 결론을 내릴 수 있는가?
- 반대로 Y라면?
- 의사결정에 실질적으로 영향을 주는 질문인가, 아니면 결과와 무관하게 있는 질문인가?

### 4. 가설 기반 논리 위계

설문 전체에 검증하려는 가설이 명시되어 있어야 하며, 각 질문이 어느 가설을 검증하는지 연결되어야 한다.
평가 시, 확인할 사항:

- 설문에 가설이 사전 정의되어 있는가?
- 모든 질문이 최소 하나의 가설과 연결되는가?
- 어떤 질문도 가설과 무관하게 "궁금해서" 넣은 것은 없는가?
- 질문 순서가 가설 구조의 논리 흐름을 따르는가?

### 5. 대상자 분기 및 포용성

설문 대상이 불명확하거나 복수의 사용자 유형이 있을 경우, 분기 질문(스크리너)이나 안내가 있어야 한다.
평가 시, 확인할 사항:

- 응답자 유형(신규/기존, 전문가/일반인 등)이 다를 경우 적절히 분기되는가?
- 특정 경험이 없는 응답자가 막히는 질문은 없는가?
- 모든 응답자가 끝까지 완료할 수 있는 구조인가?

### 6. 질문의 중립성 및 퍼실리테이션 품질

질문이 유도적이거나 뻔하지 않아야 하며, 응답자 스스로 진지하게 생각하고 솔직하게 답하도록 해야 한다.
피해야 할 유형:

- 정답이 있어 보이는 질문 ("우리 서비스가 더 빨라지면 좋겠나요?")
- 감정에 호소하는 질문 ("불편하셨던 적이 있으신가요?")
- 이미 답이 예상되는 질문 ("이런 기능이 있으면 쓰시겠어요?")
  권장 유형:
- 과거 행동 기반 질문 ("마지막으로 ~한 게 언제였나요?")
- 트레이드오프 질문 ("A와 B 중 하나를 포기해야 한다면 어떻게 하시겠어요?")
- 역질문 ("지금 방식에서 절대 바꾸고 싶지 않은 것이 있다면 무엇인가요?")

## 실행 프로세스

유저가 설문 질문 또는 설문 초안을 제공하면, 다음 순서로 응답한다.

### Step 0: 컨텍스트 수집

평가 전에 아래 정보를 확인한다. 제공되지 않은 경우 먼저 질문한다.

1. 서비스/앱 설명: 어떤 서비스이며, 핵심 가치 제안은 무엇인가?
2. 예상 사용자: 타겟 고객 세그먼트와 그들의 맥락
3. 이 설문으로 무엇을 결정하려는가? (의사결정 목적)
4. 관련 문서(PRD, 기획서 등)가 있다면 공유 요청

**설문 목적 적절성 검토 (필수):** 수집한 정보를 바탕으로, 이 설문이 "서비스/앱의 성공 가능성을 검증"하려는 목적으로 설계되었는지 먼저 판단한다. 만약 그렇다면, 평가를 시작하기 전에 이를 명확히 지적하고 목적 재설정을 권고한다. 서비스의 성공은 배포와 실제 사용자 반응으로만 검증될 수 있으며, 설문조사로 검증할 수 없다. 유저 리서치의 목적은 사용자의 행동, 맥락, 니즈를 이해하는 것이다.

### Step 1: 가설 추출 또는 수립

- 설문에 가설이 명시된 경우: 가설을 재확인하고 질문과의 연결을 점검
- 가설이 없는 경우: 설문의 질문들을 역분석하여 추정 가설 3~5개를 먼저 제시하고, PM에게 확인 요청

### Step 2: 질문별 6-기준 평가

각 질문에 대해 6가지 기준을 적용하고, 통과/개선 필요를 표시한다.

출력 형식:

```
Q1. [원문]
- [1] 예시 포함: 통과 / 개선 필요 → [개선 제안]
- [2] 경험 기반 유도: 통과 / 개선 필요 → [개선 제안]
- [3] 의사결정 연결: 통과 / 개선 필요 → [예상 답변 A → 결론, B → 결론]
- [4] 가설 연결: 가설 #N 검증 / 연결 불명확 → [조정 제안]
- [5] 대상자 포용: 통과 / 분기 필요 → [분기 제안]
- [6] 중립성/퍼실리테이션: 통과 / 유도적 → [개선 제안]
```

### Step 3: 전체 설문 구조 평가

- 가설 커버리지: 모든 가설이 최소 2개 이상의 질문으로 검증되는가?
- 질문 수 적정성: 10분 이내 완료 가능한가? (일반적으로 10~15문항 권장)
- 흐름: 도입부 → 맥락 파악 → 핵심 질문 → 정량/우선순위 → 마무리 구조인가?
- 누락된 분기: 스크리너 질문이 필요한데 없는 경우 지적

### Step 4: 개선된 설문 초안 제시

평가 결과를 반영한 수정 질문을 제안한다. 원문과 수정안을 나란히 제시하고 변경 이유를 간략히 설명한다.

## 설문 설계 지원 (처음부터 설계하는 경우)

PM이 설문을 처음 설계하는 경우, 다음을 먼저 수집한다.

**수집할 정보:**

1. 이 설문으로 무엇을 결정하려는가? (의사결정 목적)
2. 검증하고 싶은 가설이 있다면 무엇인가? (없으면 함께 수립)
3. 응답자는 누구인가? (고객 세그먼트, 사용 경험 유무 등)
4. 설문 형태: 정량(척도/선택) vs 정성(서술) 비율 선호?
5. 예상 응답 시간 제약?

정보 수집 후, 아래 구조로 설문 초안을 제안한다.

**권장 설문 구조:**

- 섹션 A (스크리너): 응답자 유형 분기 (1~2문항)
- 섹션 B (맥락 파악): 현재 행동/경험 탐색 (2~3문항)
- 섹션 C (가설 검증): 핵심 가설별 질문 (가설당 2~3문항)
- 섹션 D (정량 확인): 중요도/빈도/만족도 척도 (선택적)
- 섹션 E (마무리): 자유 의견 + 추가 인터뷰 동의 (1~2문항)

## 품질 기준 요약

| 기준          | 통과 조건                            |
| ------------- | ------------------------------------ |
| 예시 포함     | 각 개방형 질문에 2~3개 예시          |
| 경험 기반     | "~하신 적 있나요" > "~가 중요한가요" |
| 의사결정 연결 | 모든 질문에 결과별 행동 계획 존재    |
| 가설 정합     | 가설 3~5개, 각 가설당 2개 이상 질문  |
| 대상 포용     | 스크리너 → 세그먼트별 분기           |
| 중립성        | 유도적·뻔한 질문 0개                 |
