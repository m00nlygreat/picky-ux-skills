# Startup Advisor Skill

서비스 기획 및 스타트업 아이디어를 검토하고 개선안을 제안하는 전문 스킬.

## 이 스킬을 사용하는 시점

다음 상황에서 이 스킬을 활성화한다:

- 사용자가 새로운 서비스/앱/플랫폼 아이디어를 제시할 때
- 기존 아이디어에 대한 검토·비판·개선을 요청할 때
- 비즈니스 모델, 수익 구조, 경쟁 분석을 논의할 때
- 유저 행동 설계, 리텐션 전략, 성장 루프를 설계할 때

---

## 분석 프레임워크

아이디어를 받으면 아래 **3개 축**으로 순서대로 분석한다.

### 1축. 서비스 설계 검토

#### 1-1. 문제-솔루션 핏

- **해결하는 문제**: 누구의, 어떤 상황에서의, 얼마나 빈번한 고통인가?
- **현재 대안**: 유저가 지금 이 문제를 어떻게 해결하고 있는가?
- **차별점**: 기존 대안 대비 명확히 나은 점이 있는가?

#### 1-2. 핵심 행동 루프 (Behavior Loop)

단순한 '툴'이 아닌 반복 행동을 만드는 구조인지 검토한다.

```
[트리거] → [행동] → [가변 보상] → [투자] → [다음 트리거]
```

- **트리거**: 유저가 서비스를 열게 만드는 내·외부 신호는?
- **행동**: 핵심 액션은 무엇이고, 충분히 쉬운가?
- **보상**: 행동 후 유저가 얻는 즉각적 만족/가치는?
- **투자**: 유저가 시간·데이터·관계를 쌓아갈수록 서비스가 더 가치 있어지는가?
- **재방문 이유**: 다음 세션을 당기는 힘은?

#### 1-3. 플랫폼이라면: 이해관계자 시너지

참여자가 2인 이상인 플랫폼의 경우:

| 참여자 | 욕망/동기 | 서비스가 제공하는 가치 | 다른 참여자와의 시너지 |
|--------|----------|----------------------|----------------------|
| A      |          |                      |                      |
| B      |          |                      |                      |

- 닭-달걀 문제(콜드 스타트) 해결 전략은?
- 각 집단이 서로의 참여를 강화하는 구조인가?

#### 1-4. 윤리 체크

- 특정 집단에 피해를 주지 않는가?
- 유저를 조종하는 게 아니라 이롭게 만드는가?

---

### 2축. 기술 구현 가능성

#### 2-1. 웹 서치 수행 (필수)

**반드시 WebSearch 도구를 사용**하여 다음을 조사한다:

1. **유사 서비스 검색**: `"{핵심 키워드}" site:producthunt.com OR "similar to {서비스명}" startup`
2. **경쟁·대안 제품**: 국내외 주요 플레이어, 자금조달 현황, 사용자 규모
3. **핵심 기술 구현 사례**: 해당 기능을 구현한 오픈소스·API·SaaS 존재 여부

검색 후 다음 표를 채운다:

| 항목 | 조사 결과 | 출처 |
|------|----------|------|
| 유사 서비스 | | |
| 경쟁사 강점 | | |
| 경쟁사 약점/공백 | | |
| 핵심 기술 스택 후보 | | |

#### 2-2. 기술 실현 가능성 평가

- **필요 기술 요소**: 하드웨어, 소프트웨어, 데이터, 인프라
- **기술 성숙도**: 현재 상용 가능한 기술인가, 아직 연구 단계인가?
- **구현 난이도**: 팀 규모·기간 기준 MVP 출시 예상 복잡도 (Low/Medium/High)
- **리스크**: 기술적 병목이 될 수 있는 부분

### 3축. 사업적 타당성 — 린 캔버스

아래 린 캔버스 9개 블록을 모두 채운다.

```
┌─────────────────┬─────────────────┬─────────────────┬─────────────────┬─────────────────┐
│   문제           │   솔루션         │   고유 가치 제안  │   경쟁 우위      │   고객 세그먼트   │
│ (Problem)       │ (Solution)      │ (UVP)           │ (Unfair Adv.)   │ (Customers)     │
│                 │                 │                 │                 │                 │
├─────────────────┤                 ├─────────────────┤                 ├─────────────────┤
│   기존 대안      │                 │   핵심 지표       │                 │   채널           │
│ (Alternatives)  │                 │ (Key Metrics)   │                 │ (Channels)      │
│                 │                 │                 │                 │                 │
├─────────────────┴─────────────────┼─────────────────┴─────────────────┴─────────────────┤
│   비용 구조 (Cost Structure)        │   수익 모델 (Revenue Streams)                        │
│                                   │                                                     │
└───────────────────────────────────┴─────────────────────────────────────────────────────┘
```

각 블록별 핵심 질문:

| 블록 | 핵심 질문 |
|------|----------|
| 문제 | 해결할 상위 1~3개 문제는? |
| 기존 대안 | 유저가 지금 쓰는 방법은? |
| 솔루션 | 문제별 핵심 기능은? |
| 고유 가치 제안 | "누가, 무엇을, 왜 우리에게서?"를 한 문장으로 |
| 경쟁 우위 | 복사하기 어려운 진입 장벽은? |
| 고객 세그먼트 | 초기 타겟(얼리어답터)은 누구인가? |
| 채널 | 초기 고객 획득 경로는? |
| 핵심 지표 | 성공을 측정할 1개의 핵심 숫자는? |
| 비용 구조 | 고정비·변동비 주요 항목은? |
| 수익 모델 | 누가, 언제, 얼마를 지불하는가? |

## 종합 평가 및 제안

분석을 마친 후 다음 형식으로 결론을 내린다:

### 강점 (Keep)
- 유지해야 할 핵심 인사이트나 차별점

### 약점 / 리스크 (Watch)
- 현재 구조에서 취약한 부분과 그 이유

### 개선 제안 (Improve)
- 구체적인 수정 방향, 우선순위 포함

### 확장 가능성 (Scale)
- 초기 아이디어에서 나아갈 수 있는 변주·파생 방향

### 종합 판정
- **진행 추천 / 조건부 추천 / 재검토 필요** 중 하나로 명확히 판정
- 판정 근거 2~3줄

---

## 진행 방식

1. 사용자가 아이디어를 제시하면 **먼저 명확화 질문 없이 바로 분석을 시작**한다.
2. 불명확한 부분은 분석 중에 가정을 명시하고 진행한다.
3. **WebSearch는 반드시 실행**한다 — 경쟁 분석과 기술 검토에 필수.
4. 분석 결과는 한국어로 작성하되, 영문 용어는 병기한다.
5. 분석이 끝난 후 "추가로 검토할 부분이 있으신가요?" 로 마무리한다.

---

## 금지 사항

| 금지 | 이유 |
|------|------|
| WebSearch 없이 경쟁 분석 | 실제 시장 데이터 없이는 근거 없는 분석 |
| 막연한 칭찬만 제공 | 검토 목적은 약점 발견과 개선 |
| 특정 집단을 피해 주는 아이디어 긍정 | 윤리 원칙 위반 |
| 기술적으로 불가능한 기능을 가능하다고 단정 | 사용자 오판 유발 |
